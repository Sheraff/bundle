# Scenario-Centered UI Redesign Plan

## Status

Accepted product/UI direction after critique passes.

This plan supersedes the previous UI-shape discussion where the app was treated as a repository dashboard with broad filters. The product has not shipped, so route shape, page hierarchy, labels, and component structure can change without backward-compatibility constraints.

## Goal

Redesign Chunk Scope's UI from scratch so it becomes understandable, visually rich, PR-useful, and faithful to the actual product model.

The UI should not expose the storage/query model as a row of filters like:

```text
scenario + env + entrypoint + lens + metric + tab
```

Instead, the product should help users move through this user-facing model:

```text
Scenario -> Output -> What's counted -> Size -> Evidence
```

And support these jobs:

- See whether a PR or release candidate is acceptable from a bundle-policy perspective.
- Understand which scenarios and outputs changed.
- Understand what byte scope is being measured.
- Inspect history across branches/runs.
- Compare two available measured artifacts.
- Drill into expert visualizations when needed.
- Configure scenario-scoped policies.

## Constraints And Corrections

These constraints came from reviewing the existing code, plans, and product model.

### Scenarios Are Real Product Objects

Do not replace scenarios with concepts like "protected imports".

A scenario is the first-class user-defined thing being bundled and tracked.

User-facing definition:

```text
A scenario is one reproducible bundle target your team tracks over time, such as a page, app build, SSR bundle, library entry, or synthetic export.
```

Scenario sources:

- Captured build scenario: produced by a normal Vite build using the Chunk Scope plugin.
- Synthetic export scenario: generated by the GitHub Action from a small export module and bundled through Vite using the same plugin.

Synthetic export scenarios are still scenarios. They are not fake data.

### One Scenario Run Can Produce Many Outputs

A single Vite build can produce:

- Multiple arbitrary environments under Vite's Environment API.
- Multiple entrypoints/outputs within each environment.

Examples:

- `client / index.html`
- `ssr / entry-server`
- `client / export-button`
- `client / dynamic-route-admin`

Environment labels are arbitrary. If Vite does not provide one, the product uses `default`.

### Lens Is A Required Product Dimension

Different users want different byte-counting semantics.

Examples:

- `Entry JS + direct CSS`
- `All JavaScript`
- `HTML + main JS`
- `Initial JS + CSS`

Internally this is `lens`. In the UI it should usually be called:

```text
What's counted
```

Charts and expert visualizations must not mix lenses in the same plotted series.

### Size Is Separate From Lens

`raw`, `gzip`, and `brotli` are size metrics/units, not lenses.

UI label:

```text
Size
```

Values:

```text
raw
gzip
brotli
```

Store and compare exact byte values. Format only for display.

### Comparable Measurement Key

The internal comparable subject is:

```text
scenario + environment + entrypoint + lens
```

The UI should expose that as:

```text
Scenario + Output + What's counted
```

The selected `Size` controls which metric is displayed/evaluated.

### Policies Attach To Scenarios

Policies attach to scenarios and can scope down to output/lens/size.

V1 policy scope should be bounded to:

- scenario
- optional environment
- optional entrypoint
- optional `What's counted`
- required size metric

Asset-class policies should be deferred until comparison aggregates support them explicitly.

### Mini Data Viz Is Required

Mini data visualization is not optional.

The UI should contain compact visual summaries on primary cards/rows, but they must answer a question and must not replace exact numbers.

### Bundling Is Deterministic

Do not introduce a generic `confidence` concept.

Use deterministic compatibility/completeness states instead:

- measured
- incomplete
- stale
- missing baseline
- failed build
- failed upload
- no policy
- policy not applicable
- unsupported artifact

### Sourcemaps Are Out Of V1

V1 should not require sourcemaps.

The plugin artifact intentionally preserves chunks, assets, modules, imports, dynamic imports, sizes, and manifest relationships without source maps.

Sourcemaps should only be added later if the product needs original source file/line attribution after transforms/minification. They are much larger artifacts and an operational-complexity cliff.

V1 attribution labels should be honest:

- asset
- chunk
- module
- package
- unknown
- unavailable

## User-Facing Terminology

| Internal concept | UI label | Notes |
| --- | --- | --- |
| scenario | Scenario | Primary object. |
| environment + entrypoint | Output | A build target we can measure, e.g. `client / index.html`. |
| lens | What's counted | Defines which bytes are included. |
| raw/gzip/brotli | Size | Selected byte metric. |
| series | Avoid in UI | Internal comparable subject. |
| metric | Avoid or use carefully | Prefer `Size` for raw/gzip/brotli. |

Example context sentence:

```text
Reviewing PR #42 · Marketing app · client / index.html · Entry JS + direct CSS · gzip · main -> head
```

For less expert contexts, show a shorter line first and reveal the full measurement context as metadata:

```text
Reviewing PR #42 for Marketing app
client / index.html · Entry JS + direct CSS · gzip · main -> head
```

## Navigation

Primary navigation:

```text
Scenarios
Review
Compare
History
Policies
```

Rationale:

- `Scenarios` is the primary object inventory.
- `Review` is the PR/release decision mode.
- `Compare` is the explicit base/head comparison mode.
- `History` is the time/evolution mode.
- `Policies` is scenario-scoped governance.

Expert visualizer is not top-level navigation. It opens contextually from scenario/output rows.

## Core UI Loop

The app should make this loop obvious:

```text
Pick a scenario
Inspect its outputs
Choose what's counted and size
Review change/history/policy state
Drill into expert evidence when needed
```

Review, Compare, History, Policies, and Expert Visualizer are modes around this same model, not separate dashboard universes.

## Page Plan

### 1. Scenarios Home

Default app landing page.

Purpose:

- Show the inventory of tracked scenarios.
- Make it obvious what needs attention.
- Provide first-run setup guidance.

Groups:

- Needs attention
- Recently changed
- Uncovered / no policy
- Stale / missing
- Healthy

Each scenario card/row shows:

- Scenario name.
- Source/kind.
- Policy coverage.
- Latest status.
- Count of environments.
- Count of outputs.
- Count of byte-counting modes.
- Last run.
- Exactly one labeled mini-viz.

Density:

- Card mode for visual browsing.
- Compact/list mode for repositories with many scenarios.

First-run empty state:

```text
Scenarios define what Chunk Scope tracks.

Start by using the Vite plugin to capture real build scenarios, or create a synthetic export scenario from a small export module.
```

Setup flow labels:

- Detected
- Selected
- Configured

Suggested first-run flow:

```text
Connect repo
Detect captured Vite scenarios
Add synthetic export scenarios
Choose scenarios to track
Confirm outputs
Choose default What's counted
Optionally add policy
Run first measurement
```

### 2. Scenario Page

Canonical object page for one scenario.

Purpose:

- Show current state.
- Show all outputs the scenario produced.
- Show history for selected byte-counting semantics.
- Link to compare and expert visualizations.
- Show policies in context.

Default page structure:

```text
Scenario summary
Recommended next action
Current outputs
History
Policies
Expert actions
```

Top section:

- Scenario name.
- Source/kind.
- Policy summary.
- Latest run completeness.
- Recommended next action.

Current Outputs section:

- Group by environment.
- Within environment, rows for entrypoints/outputs.
- Default row columns should stay compact:
  - Output
  - Current
  - Delta
  - Policy state
  - Mini-viz
  - Primary action

Expanded row/detail shows:

- What's counted.
- Size.
- Baseline.
- Measurement state.
- Exact policy scope.
- Evidence availability.
- Expert links.

Scope controls:

- Branch/ref selector.
- What's counted selector.
- Size selector.

Output selection should live in the output rows, not as a hidden global dropdown.

History module:

- Below the current outputs, not a competing side panel by default.
- Fixed `What's counted` and `Size`.
- Main branch by default.
- Additional branches can be added.
- Line count capped; overflow switches to small multiples or explicit selection.
- Missing data appears as gaps, not zeros.

Actions:

- Review latest PR impact.
- Compare.
- Open treemap.
- Open waterfall.
- Open module graph.
- Open assets/packages tables.

### 3. Review Mode

Decision surface for a PR or release review.

Purpose:

- Answer whether the PR/release is acceptable from a bundle-policy perspective.
- Explain why.
- Show top affected scenarios and outputs.
- Provide actions.

Structure:

```text
Verdict hero
Why this verdict
Top affected scenarios
Scenario groups
Selected output evidence
Actions
```

Verdict precedence:

```text
measurement failed
incomplete
missing baseline
blocked policy
needs decision
no policy
pass
```

Separate data state from policy state.

Measurement state examples:

- complete
- failed
- incomplete
- missing baseline
- incompatible

Policy state examples:

- block
- warn
- pass
- no policy
- disabled
- not applicable
- accepted

Rows use the same scenario/output model, but Review Mode is not table-first.

Default hierarchy:

- Verdict first.
- Top 3-5 risks expanded.
- Other scenario groups collapsed with meaningful summaries.
- Detailed rows visible inside expanded groups.

Row actions:

- Accept / acknowledge where applicable.
- Open evidence.
- Compare.
- Open policy.

Release readiness should be a Review preset:

```text
release candidate vs last release
release candidate vs main
release candidate vs tag
```

This avoids hiding release prep inside generic history.

### 4. Compare Mode

Explicit comparison mode for two available measured artifacts/runs/refs.

Purpose:

- Compare PR base/head, current/main, release candidate/last release, run/run, or advanced selected artifacts.
- Show exact and non-exact comparison compatibility.
- Drill into evidence.

Presets first:

- PR base vs head
- Current vs main
- Release candidate vs last release
- Run vs run

Advanced compare exists, but should not be the default entry.

Compatibility labels:

- Exact
- Partial
- Exploratory
- Invalid

Policy-grade comparisons require exact compatibility.

Partial/exploratory comparisons are advisory.

Compatibility explainer should show dimensions:

- scenario
- environment
- entrypoint
- What's counted
- Size
- build/config identity
- artifact availability

Rows classify as:

- same
- added
- removed
- unavailable
- unsupported lens
- missing size

Current implementation note:

- Added/removed rows require a union base/head read model.
- Do not claim removed support until that exists.

### 5. History Mode

Time/evolution surface.

Purpose:

- Show scenario and repository trends.
- Support branch/tag/release markers where data exists.
- Link back to scenario and expert evidence.

Rules:

- Starts from scenario selection and fixed `What's counted` + `Size`.
- Main branch by default.
- Can show selected branch or multiple branches.
- Missing points are gaps, not zeroes.
- History key is scenario + environment + entrypoint + lens, with selected size as the displayed/evaluated metric.

### 6. Expert Visualizer

Contextual analysis mode.

Not top-level navigation.

Opened from a specific scenario/output row.

Header:

```text
Marketing app · client / index.html · Entry JS + direct CSS · gzip · main -> PR #42
```

Header context should be editable through chips, but should not act like a global filter strip.

Intent rail:

| User question | View |
| --- | --- |
| Where size lives | Treemap |
| What changed | Bundle waterfall |
| How modules connect | Module graph |
| Which assets/packages changed | Tables |

History scrubber:

- Works for the same output + `What's counted` over valid runs.
- Can power treemap/waterfall/module graph when data exists.

Compare overlay:

- Available when base/head context exists.

Compatibility banner:

- full attribution
- partial attribution
- not applicable

No sourcemaps in v1. Do not imply source-line attribution.

### 7. Policies

Scenario-first governance surface.

Policy builder should read like a sentence:

```text
Block if gzip size for Entry JS + direct CSS on client / index.html grows by more than 10 KB.
```

Policy cell consequence labels:

- Blocks merge
- Warns only
- No enforcement
- Not evaluated

Policy state should be visible in context on:

- Scenario Page
- Review Mode
- Compare Mode
- History Mode where relevant

## Mini-Viz Rules

Mini data viz is required, but must be disciplined.

General rules:

- One primary visual signal per row/card.
- Exact numbers always visible.
- No decorative visualizations.
- No misleading charts for missing/incomplete data.
- Red/green cannot be the only encoding.

Decision rule:

| Situation | Visual |
| --- | --- |
| Baseline and current exist | Delta bar |
| Policy limit exists | Delta bar with threshold marker |
| Absolute max-size policy | Small bounded gauge/bullet |
| History is the point | Sparkline |
| Run state is the point | State strip |
| Data missing/incomplete | Status chip |

Delta bars should share scale within a section, not globally across unrelated metrics.

Threshold markers require a named policy source.

Sparklines require enough comparable history.

State strips should show a capped recent history, such as the last 5-8 runs.

## Mobile And Responsive Behavior

Responsive behavior is part of the plan, not implementation cleanup.

Mobile rules:

- Scenario cards become single-column.
- Outputs matrix becomes grouped rows.
- Review top risks become cards.
- Detail drawers become bottom sheets.
- Compare uses Base / Head / Diff segmented control instead of side-by-side panes.
- Expert Visualizer uses bottom mode tabs or a command sheet.
- History scrubber stays sticky where useful.
- Expert visualizer may recommend landscape mode for dense graph views.

## Data And Contract Notes

### Shared Output Row

Build a shared row contract, not just a shared component.

It should be a discriminated union by mode:

- PR/review row
- scenario latest row
- scenario history row
- compare row
- union-pair row

Common fields:

- rowId
- seriesKey
- scenario
- environment
- entrypoint
- entrypointKind
- lens
- current totals
- baseline totals when available
- delta totals when available
- selected size
- measurementState
- policyState
- miniViz
- evidenceAvailability

Actions are derived by mode and should not be persisted on the row.

### Series Key

Current code keys series by:

```text
scenario + environment + entrypoint + lens
```

`entrypointKind` is stored but not part of the unique key today.

Decision needed before hardening contracts:

- Include `entrypointKind` in comparable identity and migrate schema, or
- Treat `entrypointKind` as metadata only and document the invariant that entrypoint keys do not collide across kinds.

### Mini-Viz Contract

`miniViz` should be typed, not an opaque UI blob.

Suggested shape:

```text
status
kind
current
baseline
delta
threshold
recentPoints
reasonWhenUnavailable
```

### Evidence Availability

Use an enum plus reason/source fields:

- available
- missing
- partial
- not_applicable
- error

### Policy State

Do not imply enforcement before real policy evaluation exists.

Initial policy state values should include:

- not_configured
- not_evaluated
- pass
- warn
- fail_non_blocking
- fail_blocking
- accepted
- disabled
- not_applicable

### Compare Union Read Model

Current compare is head-summary driven. Full Compare Mode needs base/head union pairing.

Union pairing should classify rows as:

- same
- added
- removed
- unavailable
- unsupported lens
- missing size

Do not claim full added/removed support before this read model exists.
